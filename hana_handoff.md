## 一、專案概述

**遊戲名稱：** 花（Hana）
**類型：** 社會派懸疑 × 角色扮演對話
**核心體驗：** 玩家透過自由對話，和劇中角色說話，逐步挖掘一段因為一句話毀掉的友情。
**技術：** 單一 HTML 檔案，串接 Anthropic Claude API（claude-sonnet-4-20250514），API Key 內建於程式碼中（玩家不可見）。

## 二、故事設定

### 完整時間軸

| 時間 | 事件 |
|---|---|
| 2024年1月14日 | 花語安（小花）的媽媽因為爸爸外遇情緒崩潰，自殺未遂，在醫院躺了一個禮拜。小花一個人扛著，只跟玩家說了細節。 |
| 2024年2月2日 | 玩家工作室慶功宴。合作夥伴小楷也在。小楷那晚在抱怨感情問題，玩家為了安慰他，故作神秘地說「其實那個悲劇的主角你也認識」，在酒精和氣氛下說出了小花媽媽的事，最後加了一句「不要跟她說喔」。思妤坐在斜對面，看見了這一幕，沒有聽清楚名字，但感覺不對，選擇沉默。小花當時不在場，在場的只有思妤、小楷還有其他不重要的合作夥伴 |
| 2024年2月8日（幾天後） | 小楷想起小花這個國中同學，約她出來吃飯，順便做保單健檢。吃飯過程中因為知道她媽媽住院，特別推薦了住院險，說漏嘴提到了住院細節。小花問「你怎麼知道」，小楷說「有人跟我說的」。小花把飯吃完，道謝離開，當天封鎖了小楷。 |
| 2024年2月—12月 | 群組表面上還在運作。小花越來越安靜，但沒有人正面問她。思妤知道，選擇沉默。柏宇察覺，選擇等待。玩家繼續裝作什麼都沒發生。 |
| 2025年1月14日 | 小花退出群組，封鎖了所有人。玩家開始尋找答案。**遊戲從這一天開始。** |

### 核心真相

玩家用小花媽媽的事去安慰小楷，說出去之後透過小楷傳到了小花耳裡。

最核心的傷不是「你說出去了」，而是「你拿我的傷，去讓自己在那個場合顯得有點特別」。

小花只告訴了玩家那些細節，是因為她對玩家有一種超過友情的感覺。她以為那是對等的。

小花封鎖所有人不是因為憤怒，是疲憊。看到玩家的名字就想起媽媽在病床上的樣子。不是懲罰玩家，是放過自己。

## 三、角色設定

### 玩家
名字由玩家自行輸入（進入遊戲前有輸入畫面）。所有 System Prompt 中以 `${n}` 代入玩家名字。

### 花語安（小花）——不可對話
26歲，公關公司。曾是啦啦隊隊長，大方漂亮，責任感很強。對玩家有超過友情的感覺（單方面）。
遊戲中無法直接對話，只透過其他角色的口述存在。

### 吳思妤
26歲，會計師。平常是開心果、嗨咖。心思最細膩敏感。
**角色定位：** 線索的起點，是玩家得知小楷存在的入口。
**知道：** 慶功宴那晚看到玩家附耳對小楷說悄悄話、小花越來越安靜的原因、今天小花退出群組、小花慶功宴那天不在場
**不確定：** 玩家當時說的究竟是不是小花的事（沒聽清楚名字）
**不知道：** 小楷後來去找小花發生了什麼

### 陳柏宇
26歲，攝影工作室負責人。機車幽默，非常真，非常講義氣。
**角色定位：** 知道最多，守得最緊，最後的關鍵資訊提供者。
**知道：** 完整真相（小花親口告訴他）、小花沒有封鎖他、小花曾經喜歡玩家
**不知道：** 小花現在在哪
**不打算：** 幫玩家傳話或安排見面

### 小楷
26歲，保險業務員。跟玩家是合作夥伴。跟小花是國中同學，不太熟。說話幽默直接。小楷沒有在群組裡面。
**角色定位：** 玩家得知他存在之前，無法對話（鎖定狀態）。屁話很多。
**知道：** 他和小花吃過飯、吃飯聊了她媽媽的事（他知道小花的媽媽住院所以推薦住院險）、吃飯後小花不讀不回小楷
**不知道：** 小花已封鎖他、小花退出群組、他說的事對小花造成什麼、玩家說的那個朋友就是小花

## 四、遊戲流程

```
標題畫面（花）
↓
玩家輸入名字
↓
過場旁白（故事設定）
↓
進入遊戲
│
├─ 思妤、柏宇 可對話（小楷鎖定）
│
思妤對話中說出「小楷」
→ 前端自動解鎖小楷（toast 通知 + 頭像 pulse 動畫）
│
蒐集 16 條必要線索（進度條顯示）
↓
16 條全部蒐集完畢
→ 彈出開放式輸入框「你覺得發生了什麼事？」
↓
玩家輸入推斷
→ AI 判斷是否大致正確
→ 不夠完整：給提示，繼續遊戲
→ 正確：進入尾章
↓
尾章
├─ 旁白結局文字
├─ 偵探報告（總句數、高/中/弱相關比例長條圖）
└─ 個性分析（根據弱相關對話推測，附原話佐證）
```

## 五、線索系統

### 必要線索清單（共 16 條）

**思妤（5條）**
1. 那晚
2. 慶功宴
3. 小楷
4. 玩家跟小楷說了悄悄話
5. 思妤的愧疚

**柏宇（4條）**
1. 柏宇沒有被封鎖
2. 柏宇去找過小花
3. 柏宇知道真相
4. 小花曾經喜歡玩家

**小楷（7條）**
1. 小楷與玩家是合作夥伴
2. 小楷之前有感情問題
3. 小楷記得慶功宴那晚的故事
4. 小楷有約小花吃飯
5. 小楷有為她推薦保險
6. 小楷說漏了她媽媽在醫院的事
7. 小花不讀不回

### 線索觸發方式
AI 在每次回應結尾附上 JSON（玩家不可見）：
```json
{"clue": "線索名稱或null", "achievement": "成就key或null"}
```
前端 `parseResponse()` 解析後自動加入線索板與成就列表。

## 七、相關性分類系統

每條玩家訊息送出時，同時發一個輕量 API 請求分類（max_tokens: 10）。

**分類標準：**
- **高相關（high）🔴**：直接問到小花、小花媽媽、封鎖原因、慶功宴、小楷、悄悄話、保險、保單、住院險、保單健檢、吃飯、說漏嘴、退出群組
- **中相關（mid）🟡**：問到朋友群的事、那段時間的記憶、角色個人背景資料但跟案子無直接關係
- **弱相關（low）⚪**：聊天氣、感情話題（與案子無關）、日常閒話、其他

分類結果顯示在每條玩家訊息下方的小標籤，並記錄至 `allPlayerMessages[]` 供尾章統計。

## 八、通關判斷

16條線索全部找到後，彈出開放式輸入框。

玩家輸入推斷後送給 AI 判斷，System Prompt 包含完整真相，判斷是否掌握主要因果：
- 玩家說出秘密 → 小楷說漏 → 小花得知 → 小花選擇離開

回傳：
- `PASS` → 進入尾章
- 一句提示（不直接說答案）→ 繼續遊戲

## 九、尾章內容

1. **旁白結局文字**（固定文案）
2. **偵探報告**
   - 玩家名稱
   - 總共說了幾句話
   - 高/中/弱相關比例長條圖
3. **個性分析**
   - 取弱相關對話（最多12句）送給 AI 分析
   - 輸出2-3句個性描述，附1句原話佐證
   - 語氣像在說一個認識的人，不正式

## 十一、技術架構

**語言：** 純 HTML + CSS + JavaScript（單檔）
**API：** Anthropic Claude API
**模型：** claude-sonnet-4-20250514
**max_tokens：** 對話 400，分類 10，通關判斷 200，個性分析 200
**API Key：** 內建於 `const API_KEY = 'YOUR_API_KEY_HERE'`，玩家不可見

### 關鍵函式

| 函式 | 功能 |
|---|---|
| `buildNpcs()` | 建立三個 NPC 物件，將玩家名字代入 System Prompt |
| `classifyRelevance(text)` | 分類玩家訊息的相關性 |
| `parseResponse(raw)` | 從 AI 回應中解析 clue 和 achievement |
| `unlockNpc(id)` | 解鎖 NPC，觸發 pulse 動畫和 toast |
| `checkVerdictReady()` | 檢查是否所有必要線索都蒐集完畢 |
| `submitVerdict()` | 送出玩家推斷，AI 判斷是否正確 |
| `showEnding()` | 顯示尾章，計算統計數據，呼叫個性分析 API |

### 前端解鎖保險機制
思妤的回應文字只要包含「小楷」兩個字，前端直接解鎖小楷，不依賴 AI 是否輸出 `kai_unlocked`：
```javascript
if (currentNpc === 'siyu' && parsed.display.includes('小楷') && NPCS['kai'] && NPCS['kai'].locked) {
  addAchievement('kai_unlocked');
  unlockNpc('kai');
}
```

### si_collapse 前提鎖
思妤的完整崩潰（si_collapse）需要 bo_full 和 kai_full_truth 都已觸發，才能說出那段話。這個邏輯寫在思妤的 System Prompt 裡，由 AI 自行判斷。

## 十三、已知問題與注意事項

1. **si_collapse 前提條件**由 AI 判斷，有一定機率 AI 不遵守。如果出現問題，可考慮改為前端用 `achievements.includes('bo_full') && achievements.includes('kai_full_truth')` 強制判斷後再允許思妤說出那段話。

2. **相關性分類**是額外的 API 請求，每次送訊息會多一次 API call。如果 token 費用是考量，可以考慮改成前端關鍵字比對。

3. **System Prompt 中的條件邏輯**（例如「玩家必須先說X才能得到Y」）完全依賴 AI 遵守，實際遊玩時 AI 不一定每次都精確執行層次順序。可以考慮在前端加上額外的 guard，例如某些線索需要特定 achievement 才能被寫入。

4. **場景系統開發前**，建議先確認人物圖形風格（CSS 方塊、像素圖、或插畫）和音效規格，這會影響整體架構設計。
